# Plan 1.5 — Hello-world outbound dial (design)

**Date:** 2026-05-13
**Status:** Brainstorm complete; implementation plan deferred until founder resumes.
**Predecessors:** `docs/superpowers/plans/2026-05-10-calling-foundation.md` (Plan 1, mock-mode pipeline now validated end-to-end), `docs/superpowers/research/sarvam-capabilities.md` (Sarvam ≠ managed voice bot; composite architecture needed).

## Goal

Place a real outbound call from Plivo to the founder's phone, hear "Susheela" speak one Kannada or Gujarati line synthesized by Sarvam TTS (`bulbul:v3`), then hang up. Validate:

1. Plivo trial-account dialing works against a verified destination.
2. Sarvam's Kannada **and** Gujarati TTS quality survives PSTN compression (8 kHz μ-law).
3. Signed-URL audio hosting from Supabase Storage works for an external caller (Plivo's fetcher).

This is the smallest possible proof that the underlying telephony + TTS pipes connect. It is **not** a real conversation — no STT, no LLM, no Pipecat. A single-line monologue followed by hangup.

The bilingual scope (Kannada + Gujarati in one plan) is intentional: the founder's family is bilingual; Sarvam's Gujarati TTS quality was an open question in the capability research; voice ↔ language mapping isn't documented, so picking voices for each language by ear in one session is more efficient than two passes.

## Out of scope (listed so it doesn't drift)

- Pipecat sidecar — earns its keep only when there's a real-time STT/LLM/TTS loop.
- Integration with `schedule-call` / `call-orchestrator` / `post-call-process` — those stay mock-mode. Hello-world is a parallel single-purpose path.
- The `Provider` interface in `packages/calling/` — not touched. No new provider implementation; this doesn't go through the abstraction.
- DLT / TRAI registration — not required for trial-account calls to verified destinations.
- India DID purchase — sandbox sender suffices for this scope.
- Voice cloning, MachineDetection, retries, cost guardrails, observability.
- Plivo webhook signature verification — Plivo signs its requests, but for hello-world we don't validate. Production calling will add HMAC verification (Plan 2.0).

## Components

### 1. `scripts/generate-susheela-hello.ts` (new, ~50 lines)

One-time-per-language audio generation.

**CLI:** `ts-node scripts/generate-susheela-hello.ts --lang kn|gu [--voice <id>] [--text "..."]`

**Behaviour:**

1. Read `SARVAM_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` from env.
2. Determine voice ID: default per language (Kannada: `Ritu`; Gujarati: `Priya`), overridable with `--voice`. Candidates per the research doc: {Ritu, Priya, Suhani, Kavitha, Shruti}.
3. Determine text: default greeting per language ("Namaste Ajji, idu Susheela. Hyange idira?" / Gujarati equivalent), overridable with `--text`.
4. POST to `https://api.sarvam.ai/text-to-speech` with:
   ```json
   {
     "inputs": ["<text>"],
     "target_language_code": "kn-IN" | "gu-IN",
     "speaker": "<voice>",
     "model": "bulbul:v3",
     "speech_sample_rate": 8000
   }
   ```
   The 8000 Hz rate matches PSTN. Per the Sarvam docs warning, mismatched sample rates produce artifacts.
5. Decode the base64 WAV from the response and upload to Supabase Storage:
   - Bucket: `audio-playback`
   - Key: `_susheela-hello/{lang}.{voice}.wav`
   - Service-role client (bypasses RLS).
6. Issue a 1-year signed URL via `createSignedUrl()`.
7. Print the signed URL to stdout. Operator copies it into the next step.

**Why a script, not an Edge Function:** generation is one-shot per voice/language combo. Caching the audio in Storage makes retried calls free; on-demand generation would add 1–2 s of silence after answer.

### 2. `supabase/functions/plivo-answer/index.ts` (new, ~30 lines)

Plivo answer-URL endpoint. Plivo GETs this when a call is answered; it returns the XML directing Plivo to play the audio and hang up.

**Behaviour:**
```
GET /functions/v1/plivo-answer?lang=kn|gu
→ 200 application/xml
  <Response><Play>{AUDIO_HELLO_URL_$LANG}</Play></Response>
```

Audio URLs are baked in via Edge Function secrets (`AUDIO_HELLO_URL_KN`, `AUDIO_HELLO_URL_GU`) set after step 1.

**Config:** `verify_jwt = false` in `supabase/config.toml` (added to the existing `[functions.*]` block) because Plivo's fetch is unauthenticated.

### 3. `scripts/dial-hello.ts` (new, ~30 lines)

The trigger.

**CLI:** `ts-node scripts/dial-hello.ts --to <e164> --lang kn|gu [--record true]`

**Behaviour:**

1. Read `PLIVO_AUTH_ID`, `PLIVO_AUTH_TOKEN`, `PLIVO_SANDBOX_FROM`, `SUPABASE_PROJECT_REF` from env.
2. POST to `https://api.plivo.com/v1/Account/{PLIVO_AUTH_ID}/Call/` with Basic Auth (`PLIVO_AUTH_ID:PLIVO_AUTH_TOKEN`):
   ```json
   {
     "from": "<PLIVO_SANDBOX_FROM>",
     "to": "<to>",
     "answer_url": "https://<PROJECT_REF>.supabase.co/functions/v1/plivo-answer?lang=<lang>",
     "answer_method": "GET",
     "record": <record>           // default false; only true when --record true passed
   }
   ```
3. Print the call UUID returned.

## Configuration

New env vars (also added to `.env.example` as empty placeholders):

| Variable | Where | Purpose |
|---|---|---|
| `PLIVO_AUTH_ID` | local `.env` | Plivo account auth ID, from Plivo console |
| `PLIVO_AUTH_TOKEN` | local `.env` | Plivo account auth token, from Plivo console |
| `PLIVO_SANDBOX_FROM` | local `.env` | Plivo sandbox sender ID assigned at signup |
| `SARVAM_API_KEY` | local `.env` | Already declared in `.env.example`; currently empty — populate now |
| `AUDIO_HELLO_URL_KN` | Edge Function secret | Signed URL from step 1, Kannada |
| `AUDIO_HELLO_URL_GU` | Edge Function secret | Signed URL from step 1, Gujarati |

No new database migration. No change to `audio-playback` bucket policy (signed URLs work despite the bucket being private).

## Operational flow

```
1. Plivo signup → verify your destination phone in console → record AUTH_ID, AUTH_TOKEN, sandbox From number
2. Add the 3 Plivo vars + populate SARVAM_API_KEY in .env
3. npx ts-node scripts/generate-susheela-hello.ts --lang kn --voice Ritu
     → prints AUDIO_URL_KN; copy it
4. npx ts-node scripts/generate-susheela-hello.ts --lang gu --voice Priya
     → prints AUDIO_URL_GU; copy it
5. supabase secrets set --project-ref $SUPABASE_PROJECT_REF \
     AUDIO_HELLO_URL_KN="<kn url>" AUDIO_HELLO_URL_GU="<gu url>"
6. supabase functions deploy plivo-answer --project-ref $SUPABASE_PROJECT_REF
7. npx ts-node scripts/dial-hello.ts --to +91XXXXX --lang kn
     → your phone rings, Susheela speaks in Kannada
8. Repeat step 7 with --lang gu
9. If a voice sounds wrong, swap --voice in step 3/4 and re-run.
   A new signed URL is issued each regen (URL changes even if path is identical),
   so steps 5 + 6 must run again. The Edge Function code itself doesn't change.
```

## Testing approach

No automated tests — manual smoke. Pass criteria:

- Phone rings within ~10 s of step 7.
- Audio is audibly Susheela in the right language.
- Sample rate matches PSTN — no chipmunk or slow-motion artifact. The explicit `speech_sample_rate: 8000` protects against this.
- Call hangs up cleanly when audio ends (Plivo's implicit hangup after `<Play>`).

**Optional verification:** rerun with `--record true` and listen to the Plivo recording to compare what PSTN delivered vs. what Sarvam produced. This is the same audio-fidelity check needed for voice-cloning eligibility in Phase 2.

1. **Voice ID selection.** Sarvam's voice ↔ language mapping isn't documented. Candidate set for warm/female/elder-coded: {Ritu, Priya, Suhani, Kavitha, Shruti}. Pick by ear during steps 7–9. Document chosen voices in `docs/superpowers/research/sarvam-capabilities.md` after a successful test.
2. **Sarvam Gujarati quality.** Gujarati TTS quality from `bulbul:v3` is unverified for grandmother-coded warmth. The hello-world dial is the verification. If quality is poor in Gujarati, escalate to Sarvam sales for voice recommendations (already on the action-items list in the capabilities research).
3. **Plivo India number purchase.** Deferred. Only needed when scaling past trial-account-verified destinations. Re-evaluate as part of Plan 2.0.
